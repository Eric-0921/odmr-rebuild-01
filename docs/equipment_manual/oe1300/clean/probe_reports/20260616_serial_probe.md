## 1. 设备身份

```
*IDN? -> SSI LIA-OE1351,SN:L2092228,Version:V1.3230310
```

**注意**：设备型号为 **OE1351**，属于 OE1300 系列（手册标题为 OE1300 series）。

## 2. `RALL?` 实测结果

### 2.1 返回格式

- **长度**：约 522–524 字节
- **格式**：ASCII 浮点 CSV
- **字段数**：37 个值
- **终止符**：`\r`
- **示例**：

```
-6.8591155e-05,2.5672501e-02,2.5672593e-02,9.0153079e+01,
-4.3860876e-05,2.5672555e-02,2.5672593e-02,9.0097886e+01,
...
5.3515474e-07,5.5830351e-04,1.023181539e-06,6.2500000e-04,6.2500000e-04
```

### 2.2 字段解析

37 个字段与手册表一致：

| 编码 | 参数 | 实测值（示例） |
|---:|---|---|
| 0 | X | -6.859e-05 |
| 1 | Y | 2.567e-02 |
| 2 | R | 2.567e-02 |
| 3 | θ | 9.015e+01 |
| 4–31 | XD1–θD7 | … |
| 32 | X-Noise | 5.352e-07 |
| 33 | Y-Noise | 5.583e-04 |
| 34 | Frequency | 1.023e-06 |
| 35 | AUX-IN1 | 6.250e-04 |
| 36 | AUX-IN2 | 6.250e-04 |

### 2.3 关键发现

- **没有 50 点历史缓冲**：每个参数只返回 1 个当前值。
- **没有 packet counter**：帧内无连续性标识字节。
- **帧长度接近固定**：522–524 字节，波动来自数值精度（科学计数法位数变化）。

## 3. 响应时间（修正）

> ⚠️ 早期手工探测显示 ~1.5 s/命令，但后续使用动态读取（读到 `\r` 终止符即返回）的脚本复测发现：**实际命令往返时间约为 20–80 ms**。早期 1.5 s 是读取策略（等超时）造成的假象。

### 3.1 单条命令实测

| 指令 | 往返时间 | 返回字节数 |
|------|---------|-----------|
| `*IDN?` | ~26 ms | 46 |
| `FMOD?` | ~23 ms | 1 |
| `FREQ?` | ~26 ms | 16 |
| `PHAS?` | ~26 ms | 13 |
| `SNAP? 0` | ~23 ms | 15 |
| `RALL?` | ~74 ms | ~522 |
| `*RST` | ~3000 ms | 多行 boot 信息 |
| `NMOD?` / `NIPA?` / `NSMA?` / `NGWA?` | ~3000 ms | 超时无响应 |

### 3.2 连续 `RALL?` 爆发测试

100 次连续 `RALL?`（命令间约 1 ms 间隔）：

- 总时间：5.504 s
- 平均吞吐：**18.17 Hz**
- 单次延迟：min 45.16 / mean 53.59 / median 47.10 / max 62.71 ms

**结论**：OE1351 串口实际采集能力约为 **18 Hz**，远高于早期估计的 0.5–1 Hz。

## 4. 对项目的含义

### 4.1 与 OE1022D 对比

| 项目 | OE1022D | OE1351（OE1300 系列） |
|------|---------|----------------------|
| `RALL?` 周期 | ~50 ms | **~50 ms** |
| 连续采集吞吐 | ~20 Hz | **~18 Hz** |
| 返回格式 | 12288B 二进制，50 点/字段 | 522B ASCII，1 点/字段 |
| 数据吞吐 | 高 | 低 |
| 连续性审计 | `payload[12287]` packet counter | 无 |
| 串口参数配置 | 支持 | **基本不支持**（除 `NMOD 0` 外）|

### 4.2 开发影响

- **串口采集频率上限约为 18 Hz**，接近 OE1022D 的 20 Hz，基本可满足多数 ODMR 实验需求。
- 网口路径仍值得测试，因为网口可能进一步降低延迟并支持示波器/FFT 功能（手册 3.16/3.17 节）。
- 若网口始终无法启用，**串口路径单独即可作为可行方案**。

**所有 set 命令**：`ISRC 0`、`ICPL 0`、`IGND 0`、`IRNG 2`、`FMOD 0`、`FREQ 1000`、`PHAS 0`、`RSLP 0`、`OFLT 0.01`、`OFSL 1`、`SYNC 0`、`HARM 1`、`BAUD 115200`

### 5.4 关键发现：串口基本只读

即使发送手册明确允许的 set 命令（如 `ISRC 0`、`FREQ 1000`），设备也**返回空响应，参数不生效**（再次查询仍为原值）。

**例外**：`NMOD 0` 会返回 `"Setting Ethernet Mode Success!"`，但其他网络 set 命令（`NIPA`、`NSMA`、`NGWA`、`NMAC`）同样无响应。

这说明当前固件 `V1.3230310` 的串口协议**基本处于只读模式**：可以查询状态和读取 `RALL?`，但无法通过串口修改设备参数。

### 5.5 `*RST` 行为

`*RST` 不是简单的参数清零，而是触发 **Xilinx Zynq 全系统重启**，返回 boot loader / PMU 信息，耗时约 3 s。

### 5.6 网络接口状态

网络查询命令在当前固件 `V1.3230310` 上基本无响应，说明**网口配置命令未实现或存在 bug**。

## 6. 对项目的影响

### 6.1 可用能力

- 串口连续 `RALL?` 采集：~18 Hz，37 字段 ASCII CSV。
- 状态查询：`*PLL?`、`OVLD?`、`TEMP?` 等。
- 默认配置读取：输入源、耦合、量程、参考源、滤波器等。

### 6.2 不可用能力

- **串口参数设置**：无法通过串口修改 `ISRC`、`FREQ`、`OFLT` 等运行参数。
- **网口通信**：设备默认 IP 不可达，网络配置命令不完整。
- **高级功能**：D 后缀命令、扫频、示波器/FFT 相关命令均超时。

### 6.3 第一版实现策略

OE1351 在第一版中应被视为**固定配置的只读观察者**：

1. 实验前通过前面板或 Console 软件把设备配置好。
2. C# runtime 只通过串口发送查询命令（主要是 `RALL?`）。
3. 不尝试在运行中重配置设备参数。
4. profile 中只保留与查询/采集相关的字段，set 命令字段标记为不可用。

### 6.4 后续排查方向

1. 联系厂商确认 `V1.3230310` 是否支持串口 set 命令，或是否需要升级固件。
2. 在 Windows 上用原厂 Console/LabVIEW 软件测试 set 命令是否可用（排除 Mac/串口适配器问题）。
4. OE1351 与 OE1300 在协议上是否完全一致？
