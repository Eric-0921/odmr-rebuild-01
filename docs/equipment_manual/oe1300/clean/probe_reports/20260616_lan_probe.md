## 0. OE1301 网口实测结果（重大更新）

> 本机设备：`SSI LIA-OE1301, SN:L2283261, Ver:V1.4h4-1.02-1.4.0.23B`

### 0.1 设备已持有网络配置

通过串口查询网络参数，OE1301 返回完整配置：

```text
NIPA? -> 192.168.1.1
NGWA? -> 192.168.1.1
NSMA? -> 255.255.255.0
NMAC? -> 52:AF:6E:D0:08:B5
NMOD? -> 1
```

> 注：`NMOD?` 返回 `1`，但设备实际上以静态 IP `192.168.1.1` 响应，说明 `NMOD` 的枚举值与手册可能存在差异，或该参数未实际生效。

### 0.2 TCP 连接成功

为 Mac `en5` 添加别名 `192.168.1.10/24` 后：

```bash
ping -c 4 192.168.1.1
nc -vz 192.168.1.1 10001
```

结果：

```text
PING 192.168.1.1: 0.0% packet loss, avg 0.497 ms
Connection to 192.168.1.1 port 10001 [tcp/scp-config] succeeded!
ARP: ? (192.168.1.1) at 52:af:6e:d0:8:b5 on en5 ifscope [ethernet]
```

### 0.3 TCP 命令交互

简单文本查询通过 TCP 正常返回 ASCII：

```text
*IDN?  -> SSI LIA-OE1301,SN:L2283261,Ver:V1.4h4-1.02-1.4.0.23B
*PLL?  -> 0
FREQ?  -> 6.698721094e+04
```

### 0.4 关键问题：RALL? / SNAP? 返回二进制乱码

通过 TCP 发送 `RALL?` 和 `SNAP? 0,1,2,3` 时，设备返回二进制数据，而非手册/串口上的 ASCII CSV：

| 命令 | 返回长度 | 内容 |
|------|----------|------|
| `RALL?` | 8192 B（首次读出）/ 32768 B（按 LabVIEW 的 32768 B 缓冲区读取） | 以 `0x80 00 00 00 00 00 00 00` 为主，夹杂无意义浮点；缓冲区尾部甚至出现 `*IDN?` 的字符串残留 |
| `SNAP? 0,1,2,3` | 4096 B | 同一 8 字节模式重复 512 次 |

按 400 B/帧、8 B 小端 double、37 字段固定偏移（见下节 LabVIEW 解析方案）解析后仍无物理意义；按 296 B/37-double 或其他常见帧长解析同样无效。`SNAP?` 缓冲区呈现明显的“卡死”特征。

这表明：

- **OE1301 固件的 TCP 数据查询路径存在缺陷**，未按手册返回 ASCII CSV。
- 返回的 32768 B 与 LabVIEW 驱动的读取缓冲区大小一致，但内容不符合 LabVIEW 期望的 400 B 帧格式。
- 当前固件下，**网口不适合作为 ODMR 主采集通道**。

### 0.5 LabVIEW 驱动解析方案对照

对原厂 LabVIEW 驱动 `OUTPUT-ssi_oe1351/` 中的 `OE1311&OE1351_Ethernet__Query_Data.vi` 与 `OE1311&OE1351_DATA Transmit.vi` 分析结果：

- `Ethernet__Query_Data.vi` 发送 `RALL?` 后等待 5 ms，然后循环读取 TCP，每次最多 32768 B，直到累积长度 ≥ 32767 B，再交给 `DATA Transmit.vi` 解析。
- `DATA Transmit.vi` 按 **400 B / 帧** 切分，每帧含 **50 × 8 B 小端 double**；取出 37 个字段，固定偏移从 X@0 到 AUX-IN2@288。
- 30000 B 固定缓冲区尾部：偏移 29990 为状态字节（PLL/Overload），偏移 29997 为 `Trig_Count`（U8）。

OE1301 实际返回的 32768 B 二进制数据：

- 按上述 400 B 帧解析得到的是无意义数值。
- 缓冲区尾部（约 32682 B 处）出现 `SSI LIA-OE1301,SN:L2283261,Ver:V1.4h4-1.02-1.4.0.23B` 字符串残留。
- 因此**即便使用原厂 LabVIEW 驱动，OE1301 的网口数据查询也很可能失败**；当前二进制响应不是 LabVIEW 可解析的 400 B 帧。

### 0.6 OE1301 网口结论

- 网口链路层和 TCP 10001 端口均可达。
- 文本命令（`*IDN?`、`FREQ?` 等）可用。
- `RALL?` / `SNAP?` 返回不可解析的二进制数据，且与 LabVIEW 驱动的 400 B 帧格式不匹配，**无法替代串口 ASCII 采集**。
- **环境限制说明**：当前探测基于 Mac 裸 TCP socket。原厂 LabVIEW 驱动依赖 Windows NI-VISA，Mac 没有 NI-VISA 运行时，读终止符、缓冲策略可能与原厂驱动不同。要排除“Mac 环境导致”这一变量，需在 Windows + NI-VISA（C# 主栈）上复测。
- 第一版仍应使用串口作为 ODMR 采集路径。

- **设备默认 IP**：`192.168.1.1`

- **PC 端设置**（见 `图 72`）：
  - IP 地址：`192.168.1.10`
  - 子网掩码：`255.255.255.0`

### 0.2 手册中 OE1351 的 TCP 通信实例

`图 75` 显示 NetAssist 已连接 `192.168.1.1:10001`，发送 `*IDN?` 后返回：

```text
SSI LIA-OE1351, SN:L0, Ver:V1.4e-1.01-1.2.0.22A
```

这说明：
1. **同型号 OE1351 确实可以通过 TCP/10001 通信**，且响应格式与串口一致。
2. 手册示例设备的固件版本为 `V1.4e-1.01-1.2.0.22A`，而我们实际设备为 `Version:V1.3230310`，**固件版本差异显著**。
3. 手册第 7.2 节的网口环境搭建**完全基于出厂默认 IP `192.168.1.1:10001`**，并未演示如何通过串口命令启用/配置网口。

### 0.3 为什么串口网络配置可能失败

手册命令参考中虽然存在 `NMOD`/`NIPA`/`NSMA`/`NGWA`，但：
- 第 7.2 节操作实例**没有使用这些命令**，而是直接连接默认 IP。
- 我们发送 `NMOD 0` / `NIPA 192.168.1.5` / `NGWA 192.168.1.1` 时均无响应，仅 `NSMA 255.255.255.0` 返回 `"Setting Ethernet Mode Success!"`。
- 重启后所有网络参数查询仍为空。

由此推断：
1. **串口网络配置命令在当前固件（V1.3230310）上可能未正确实现，或需要额外的使能/保存步骤**（手册未给出）。
2. **设备网口可能出厂即应处于默认 `192.168.1.1:10001` 状态**；若该默认状态不存在，说明本机网络栈/配置未初始化或硬件异常。
3. 不能排除当前固件版本网口功能存在缺陷。

## 新增关键发现：LabVIEW OE1311/OE1351 驱动

拉取远端更新后，仓库新增 `reverse_application/oe1351-labview/OUTPUT-ssi_oe1351/` 目录，包含原厂 LabVIEW VI 导出文件。其中与网口配置直接相关的 VI：

- `OE1311&OE1351__Configure_Ethernet`
- `OE1311&OE1351__Ethernet__Query_Data`
- `OE1311&OE1351__Ethernet__Query_Setting`
- `OE1311&OE1351__Configure_Reset`

### 0.1 LabVIEW 中的网口配置方式

`Configure Ethernet.vi` 的 block diagram（`OE1311&OE1351__Configure_Ethernetd.png`）显示：

1. **通过 RS232 串口发送网络配置命令**，而非通过网口自身配置。
2. 命令序列：`NMOD` → `NIPA` → `NSMA` → `NGWA` → `NMAC`。
3. 命令之间用 `;` 连接，最终一次性 VISA Write 出去。
4. **等待时间 25000 ms（25 秒）** 后再读取设备返回。

这说明网络配置命令的响应可能极慢，我们之前 3 s / 10 s 的超时可能仍然不够长。

### 0.2 LabVIEW 默认 IP 与手册不同

`Configure Ethernet.vi` 前面板默认：

- PC IP Address：`192.168.0.10`
- Subnet Mask：`255.255.255.0`
- Gateway：`192.168.0.1`
- MAC Address：`89:28:C8:64:E6:CE`

这与手册第 7.2 节的 `192.168.1.1/10001` **不一致**。LabVIEW 驱动的默认网关是 `192.168.0.1`，暗示 OE1351 出厂默认可能是 `192.168.0.1` 而非 `192.168.1.1`。

### 0.3 长超时复测结果

按 LabVIEW 方式，把单条/串接命令超时延长到 10–35 s：

```text
NMOD?                               -> timeout (10 s)
NIPA?                               -> timeout (10 s)
NSMA?                               -> timeout (10 s)
NGWA?                               -> timeout (10 s)
NMAC?                               -> timeout (10 s)
NMOD 0                              -> ok      (6 s)  'Setting Ethernet Mode Success!'
NIPA 192.168.1.5                    -> timeout (10 s)
NSMA 255.255.255.0                  -> timeout (10 s)
NGWA 192.168.1.1                    -> timeout (10 s)
NMAC 89:28:C8:64:E6:CE              -> timeout (10 s)
NMOD 0;NIPA 192.168.1.5;NSMA ...    -> ok      (6 s)  'Setting Ethernet Mode Success!'
```

结论：
- 只有 `NMOD 0` 能在约 6 s 后返回 `"Setting Ethernet Mode Success!"`。
- `NIPA`/`NSMA`/`NGWA`/`NMAC` 无论是 set 还是 query，在 10 s 内均无响应。
- 串接命令后也只得到一条成功响应，推测设备只解析了第一条 `NMOD 0`。

### 0.4 当前最可能原因

当前固件 `V1.3230310` 的网口配置命令集**不完整或存在 bug**：`NMOD` 有响应，但 IP/掩码/网关/MAC 设置命令无响应。因此即使按原厂 LabVIEW 方式操作，也无法通过串口把设备配到可用 IP。

## 8. 后续建议

由于串口配置网络未成功，建议改用以下方法：

### 8.1 使用 PC Console 软件配置（最可靠）

手册第 6 章描述 Console 软件可通过 TCP/IP 连接设备。但 LabVIEW 驱动默认网关为 `192.168.0.1`，与手册 `192.168.1.1` 冲突，建议两个子网都试：

1. 在 Windows PC 上安装 Console 软件。
2. 用网线连接 OE1351 和 PC。
3. **方案 A（手册）**：PC 网卡 `192.168.1.10/24`，网关 `192.168.1.1`，Console 连接 `192.168.1.1:10001`。
4. **方案 B（LabVIEW 驱动）**：PC 网卡 `192.168.0.10/24`，网关 `192.168.0.1`，Console 连接 `192.168.0.1:10001`。
5. 如果连接成功，可在 Console 中查看/修改网络参数。

### 8.2 检查物理状态

- 观察 OE1351 网口指示灯：连接时应有 Link 灯亮起，数据收发时应有 Activity 灯闪烁。
- 尝试更换网线（确认是直通线，千兆口一般支持 Auto-MDIX，但可排除线缆问题）。
- 尝试将 OE1351 接到交换机/路由器，而不是直连 Mac。

### 8.3 检查固件版本

当前固件版本：`V1.3230310`。可联系厂商确认该版本网口功能是否正常，或是否有升级固件。

### 8.4 放弃网口，评估串口可用性

如果网口始终调不通，则需基于串口实测结果评估 OE1351 是否可用于项目：

- 串口实测连续 `RALL?` 可达 **~18 Hz**，单条命令延迟约 20–80 ms，已基本满足 ODMR 需求。
- 若网口调通，可能进一步降低延迟并支持示波器/FFT 功能；但网口并非唯一可行路径。

## 9. 端口 10001 复测结果

在 Mac 上保持 `en5` 为手册推荐的 `192.168.1.10/24` 配置，对默认 IP 和端口 10001 进行复测：

```bash
ping -c 3 -W 1000 192.168.1.1
nc -vz -w 3 192.168.1.1 10001
arp -a | grep 192.168.1
```

结果：

```text
PING 192.168.1.1: 3 packets transmitted, 0 received, 100% packet loss
nc to 192.168.1.1:10001: failed
ARP: ? (192.168.1.1) at (incomplete) on en5 ifscope [ethernet]
```

**结论**：即使使用手册明确的默认 IP `192.168.1.1` 和固定端口 `10001`，本机 OE1351 仍在数据链路层无响应。问题不是端口或 IP 配置错误，而是设备网口未激活/未初始化，或当前固件版本不支持。

> 更新：LabVIEW 驱动默认使用 `192.168.0.1` 网关。由于 Mac 上 `192.168.1.10` 别名在一次 `*RST` 后已丢失，且 sudo 需要交互密码，**尚未在 Mac 上验证 `192.168.0.0/24` 子网**。

### 9.2 验证 LabVIEW 驱动默认子网 `192.168.0.0/24`

按 LabVIEW 驱动默认值，将 Mac `en5` 同时配置为 `192.168.0.10/24` 和 `192.168.1.10/24`，分别测试两个默认网关：

```bash
sudo ifconfig en5 alias 192.168.0.10/24 up
sudo ifconfig en5 alias 192.168.1.10/24 up

ping -c 2 -W 1000 192.168.0.1
ping -c 2 -W 1000 192.168.1.1
nc -vz -w 3 192.168.0.1 10001
nc -vz -w 3 192.168.1.1 10001
arp -a | grep -E "192\.168\.[01]"
```

结果：

```text
192.168.0.1: 2 packets transmitted, 0 received, 100% packet loss
192.168.1.1: 2 packets transmitted, 0 received, 100% packet loss
ARP:
  ? (192.168.0.1) at (incomplete) on en5 ifscope [ethernet]
  ? (192.168.1.1) at (incomplete) on en5 ifscope [ethernet]
```

**结论**：无论按手册的 `192.168.1.1` 还是按 LabVIEW 驱动的 `192.168.0.1`，本机 OE1351 都未在数据链路层响应。

### 9.3 全子网 ping 扫描

为排除设备被配置到非默认 IP 的可能，对 `192.168.1.0/24` 做并行 ping 扫描：

```bash
for ip in $(seq 1 254); do
  (ping -c 1 -W 500 -S 192.168.1.10 192.168.1.$ip > /dev/null 2>&1 && echo "RESPONSE: 192.168.1.$ip") &
done; wait
```

结果：**仅 `192.168.1.10`（Mac 自身）响应**，其余 253 个地址均无响应；ARP 表全部显示 `(incomplete)`。

这说明：
1. Mac 侧 `en5` 配置和路由是正确的，能够从 `192.168.1.10` 正常发起 ARP/ICMP。
2. OE1351 在 `192.168.1.0/24` 整个子网内都没有活跃的网络接口。
3. 设备网口要么未物理连接/未上电，要么网络栈未启动，要么被配置到了其他子网/静态 IP。

### 10.1 在 Mac 上验证两个子网 ✅ 已完成


结果：两个默认 IP 均不响应，ARP 均为 `(incomplete)`。

### 10.2 尝试恢复出厂默认网络参数

通过串口发送 `*RST` 复位设备（注意：所有运行参数也会恢复默认）：

```text
*RST
```

等待设备重启后，再次 ping `192.168.1.1` 并测试 `nc -vz 192.168.1.1 10001`。

### 10.3 检查网口指示灯与线缆

- 设备网口应有 **Link 灯常亮** 和 **Activity 灯闪烁**。
- 若 Link 灯不亮，即使 `en5` 显示 `1000baseT` 也可能是协商假象或线缆问题。
- 尝试更换网线，或把 OE1351 接到交换机/路由器观察是否能获取到 MAC 地址。

### 10.4 确认固件版本差异

当前设备版本 `V1.3230310` 与手册截图中的 `V1.4e-1.01-1.2.0.22A` 格式完全不同。建议向厂商确认：

- `V1.3230310` 是否支持网口 TCP Server 功能；
- 该版本是否需要特定命令启用网口；
- 是否存在已知 bug 或升级固件。


---

## 11. 最终结论

综合手册、原厂 LabVIEW 驱动和真机实测，当前 OE1351（固件 `V1.3230310`）的网口状态如下：

1. **Console/NetAssist 理论连接参数已确认**：TCP Client，`192.168.1.1:10001`（手册）或 `192.168.0.1:10001`（LabVIEW 驱动）。
2. **串口网络配置命令不完整**：仅 `NMOD 0` 能在约 6 s 后返回 `"Setting Ethernet Mode Success!"`；`NIPA`/`NSMA`/`NGWA`/`NMAC` 无论 set/query 均超时。
3. **两个默认子网均不可达**：Mac 已配置 `192.168.0.10/24` 和 `192.168.1.10/24`，但 `192.168.0.1` 与 `192.168.1.1` 均无 ARP/MAC 响应。
4. **最可能原因**：当前固件版本网口功能存在缺陷，或设备网口硬件/网络栈未启动。

**工程决策建议**：

- **第一版放弃网口**，基于串口实现 OE1300/1351 支持。
- 串口实测连续 `RALL?` 可达 **~18 Hz**，单命令延迟 20–80 ms，已能满足多数 ODMR 实验需求。
- 若后续需要网口，必须先联系厂商确认 `V1.3230310` 网口功能或升级固件。
