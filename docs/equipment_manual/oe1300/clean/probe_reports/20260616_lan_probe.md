# OE1300/OE1351 网口探测报告

> 探测时间：2026-06-16  
> 网络配置：Mac en5 已添加别名 `192.168.1.10/24`  
> 目标设备默认 IP：`192.168.1.1`

---

## 0. 关键更新（基于手册截图复核）

复核 `docs/equipment_manual/oe1300/clean/chapter_07.md` 中的网口通讯截图后，发现以下关键细节：

### 0.1 Console / NetAssist 实际连接参数

- **协议**：TCP Client
- **设备默认 IP**：`192.168.1.1`
- **固定端口**：`10001`（手册第 7.2 节原文："远程主机端口固定为 `10001`"）
- **PC 端设置**（见 `图 72`）：
  - IP 地址：`192.168.1.10`
  - 子网掩码：`255.255.255.0`
  - 默认网关：`192.168.1.1`（即设备 IP）

### 0.2 手册中 OE1351 的 TCP 通信实例

`图 75` 显示 NetAssist 已连接 `192.168.1.1:10001`，发送 `*IDN?` 后返回：

```text
SSI LIA-OE1351, SN:L0, Ver:V1.4e-1.01-1.2.0.22A
```

这说明：
1. **同型号 OE1351 确实可以通过 TCP/10001 通信**，且响应格式与串口一致。
2. 手册示例设备的固件版本为 `V1.4e-1.01-1.2.0.22A`，而我们实际设备为 `Version:V1.3230310`，**固件版本差异显著**。
3. 手册第 7.2 节的网口环境搭建**完全基于出厂默认 IP `192.168.1.1:10001`**，并未演示如何通过串口命令启用/配置网口。

### 0.3 为什么串口网络配置可能失败

手册命令参考中虽然存在 `NMOD`/`NIPA`/`NSMA`/`NGWA`，但：
- 第 7.2 节操作实例**没有使用这些命令**，而是直接连接默认 IP。
- 我们发送 `NMOD 0` / `NIPA 192.168.1.5` / `NGWA 192.168.1.1` 时均无响应，仅 `NSMA 255.255.255.0` 返回 `"Setting Ethernet Mode Success!"`。
- 重启后所有网络参数查询仍为空。

由此推断：
1. **串口网络配置命令在当前固件（V1.3230310）上可能未正确实现，或需要额外的使能/保存步骤**（手册未给出）。
2. **设备网口可能出厂即应处于默认 `192.168.1.1:10001` 状态**；若该默认状态不存在，说明本机网络栈/配置未初始化或硬件异常。
3. 不能排除当前固件版本网口功能存在缺陷。

## 1. 探测结果

### 1.1 Ping 测试

```bash
ping -c 3 192.168.1.1
```

结果：100% packet loss，无响应。

### 1.2 子网扫描

扫描 `192.168.1.1` 到 `192.168.1.10`：

```bash
for ip in 192.168.1.{1..10}; do ping -c 1 -W 500 $ip; done
```

结果：只有 `192.168.1.10`（Mac 自身）响应，未发现 OE1300/OE1351。

### 1.3 串口查询网络配置

通过串口查询网络参数：

```
NMOD? -> （空）
NIPA? -> （空）
NSMA? -> （空）
NGWA? -> （空）
```

**推断**：OE1351 的网口当前未配置 IP，或未启用 TCP 服务。

---

## 2. 可能原因

1. **设备网口未配置**：串口查询返回空，说明网络参数可能是出厂未设置状态。
2. **默认 IP 不是 192.168.1.1**：虽然手册写明默认 `192.168.1.1`，但实际设备可能被修改过。
3. **网口未连接/未启用**：物理链路或设备固件中网络栈未启动。
4. **需要 crossover 线或特定网线**：但现代设备通常支持 Auto-MDIX。

---

## 3. 下一步排查建议

### 3.1 通过串口配置网络

尝试用串口发送网络配置命令：

```
NMOD 0          // 设置为 TCP 模式
NIPA 192.168.1.5
NSMA 255.255.255.0
NGWA 192.168.1.1
```

然后再次 ping 192.168.1.5 并扫描端口。

### 3.2 使用 Console 软件配置

OE1300 的 PC Console 软件可能通过串口发现并配置网口。可以运行 Console 查看设备是否出现在串口列表，并通过 GUI 配置网络。

### 3.3 检查物理连接

- 确认网线是直连设备网口，而非通过交换机。
- 尝试更换网线或端口。
- 观察设备网口指示灯状态。

---

## 4. 当前结论

- **串口**：已验证可用。早期探测误以为 ~1.5 s/命令，复测显示单条命令约 20–80 ms，连续 `RALL?` 可达 **~18 Hz**。
- **网口**：当前未成功通信，设备可能未配置或未启用网络参数。
- 在网口未调通之前，**无法判断网口是否比串口快**。

## 5. 选项 A 执行结果

通过串口向 OE1351 发送网络配置命令：

```text
NMOD 0
NIPA 192.168.1.5
NSMA 255.255.255.0
NGWA 192.168.1.1
```

响应：

```text
NSMA 255.255.255.0 -> "Setting Ethernet Mode Success!"
其他命令 -> 无响应
```

但后续查询仍返回空，且 ping 192.168.1.5 仍然超时。

### ARP 状态

```
? (192.168.1.5) at (incomplete) on en5 ifscope [ethernet]
```

OE 设备未响应 ARP 请求，说明网口在数据链路层仍未激活。

### 可能原因

1. 网络配置虽已写入，但设备需要重启/断电才能生效。
2. 设备网络栈当前未启动。
3. 网线/物理连接需要确认（虽然 en5 显示 1000baseT full-duplex active）。

## 7. 重启后复测

2026-06-16，用户重启 OE1351 后复测：

- 串口仍可用：`*IDN?` 正常返回 `SSI LIA-OE1351,...`
- 网络参数查询仍返回空：`NMOD?`、`NIPA?`、`NSMA?`、`NGWA?` 均为空
- Ping 192.168.1.5 仍然超时
- ARP 表无新 MAC 地址出现
- 子网 mDNS 浏览无结果

### 结论

通过串口发送的网络配置命令**未在设备上持久化**，或设备网口**当前未激活**。重启后配置丢失。

## 8. 后续建议

由于串口配置网络未成功，建议改用以下方法：

### 8.1 使用 PC Console 软件配置（最可靠）

手册第 6 章描述 Console 软件可通过 TCP/IP 连接设备。建议：

1. 在 Windows PC 上安装 Console 软件。
2. 用网线连接 OE1351 和 PC。
3. 将 PC 网卡设为 `192.168.1.10/24`，默认网关 `192.168.1.1`，DNS 留空。
4. 打开 Console，选择 `TCP/IP`，输入 `192.168.1.1`，点击 `Connect`。
5. Console 实际通过 **TCP Client 连接到 `192.168.1.1:10001`**。
6. 如果连接成功，说明设备默认确实在 `192.168.1.1:10001`，此时可在 Console 中查看/修改网络参数。

### 8.2 检查物理状态

- 观察 OE1351 网口指示灯：连接时应有 Link 灯亮起，数据收发时应有 Activity 灯闪烁。
- 尝试更换网线（确认是直通线，千兆口一般支持 Auto-MDIX，但可排除线缆问题）。
- 尝试将 OE1351 接到交换机/路由器，而不是直连 Mac。

### 8.3 检查固件版本

当前固件版本：`V1.3230310`。可联系厂商确认该版本网口功能是否正常，或是否有升级固件。

### 8.4 放弃网口，评估串口可用性

如果网口始终调不通，则需基于串口实测结果评估 OE1351 是否可用于项目：

- 串口实测连续 `RALL?` 可达 **~18 Hz**，单条命令延迟约 20–80 ms，已基本满足 ODMR 需求。
- 若网口调通，可能进一步降低延迟并支持示波器/FFT 功能；但网口并非唯一可行路径。

## 9. 端口 10001 复测结果

在 Mac 上保持 `en5` 为手册推荐的 `192.168.1.10/24` 配置，对默认 IP 和端口 10001 进行复测：

```bash
ping -c 3 -W 1000 192.168.1.1
nc -vz -w 3 192.168.1.1 10001
arp -a | grep 192.168.1
```

结果：

```text
PING 192.168.1.1: 3 packets transmitted, 0 received, 100% packet loss
nc to 192.168.1.1:10001: failed
ARP: ? (192.168.1.1) at (incomplete) on en5 ifscope [ethernet]
```

**结论**：即使使用手册明确的默认 IP `192.168.1.1` 和固定端口 `10001`，本机 OE1351 仍在数据链路层无响应。问题不是端口或 IP 配置错误，而是设备网口未激活/未初始化，或当前固件版本不支持。

### 9.2 全子网 ping 扫描

为排除设备被配置到非默认 IP 的可能，对 `192.168.1.0/24` 做并行 ping 扫描：

```bash
for ip in $(seq 1 254); do
  (ping -c 1 -W 500 -S 192.168.1.10 192.168.1.$ip > /dev/null 2>&1 && echo "RESPONSE: 192.168.1.$ip") &
done; wait
```

结果：**仅 `192.168.1.10`（Mac 自身）响应**，其余 253 个地址均无响应；ARP 表全部显示 `(incomplete)`。

这说明：
1. Mac 侧 `en5` 配置和路由是正确的，能够从 `192.168.1.10` 正常发起 ARP/ICMP。
2. OE1351 在 `192.168.1.0/24` 整个子网内都没有活跃的网络接口。
3. 设备网口要么未物理连接/未上电，要么网络栈未启动，要么被配置到了其他子网/静态 IP。

## 10. 基于截图复核后的补充排查建议

### 10.1 在 Mac 上验证默认端口 10001

确保 `en5` 配置与手册一致：

```bash
sudo ifconfig en5 alias 192.168.1.10/24 up
# 注意：这会与之前设置的 192.168.1.10/24 相同；若此前已设则无需重复
```

然后分别测试默认 IP 和之前尝试的 `192.168.1.5`：

```bash
nc -vz -w 3 192.168.1.1 10001
nc -vz -w 3 192.168.1.5 10001
```

同时观察 ARP 表变化：

```bash
arp -a | grep 192.168.1
```

### 10.2 尝试恢复出厂默认网络参数

通过串口发送 `*RST` 复位设备（注意：所有运行参数也会恢复默认）：

```text
*RST
```

等待设备重启后，再次 ping `192.168.1.1` 并测试 `nc -vz 192.168.1.1 10001`。

### 10.3 检查网口指示灯与线缆

- 设备网口应有 **Link 灯常亮** 和 **Activity 灯闪烁**。
- 若 Link 灯不亮，即使 `en5` 显示 `1000baseT` 也可能是协商假象或线缆问题。
- 尝试更换网线，或把 OE1351 接到交换机/路由器观察是否能获取到 MAC 地址。

### 10.4 确认固件版本差异

当前设备版本 `V1.3230310` 与手册截图中的 `V1.4e-1.01-1.2.0.22A` 格式完全不同。建议向厂商确认：

- `V1.3230310` 是否支持网口 TCP Server 功能；
- 该版本是否需要特定命令启用网口；
- 是否存在已知 bug 或升级固件。
