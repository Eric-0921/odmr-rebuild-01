# 变更与踩坑

本文集中记录当前 rebuild 已经踩实的技术事实，避免后续再掉回旧坑里。

## 1. `*OPC?` 不能单独当作 sweep 结束信号

真机已经证明：

- 当前实验室 SMB100A 上，`SWE:FREQ:EXEC` 后的 `*OPC?` 可能在几毫秒内返回
- 它会明显早于真实 sweep 结束

当前 runtime 方案：

- 先发 `*OPC?`
- 记录 `opc_wait_ms`
- 如果等待时间明显短于 `start/stop/step/dwell` 推出来的 sweep 时长
- 则退回到：
  - `estimated sweep duration + guard`

这不是临时 workaround，而是当前主运行链的一部分。

## 2. sweep 时间估算不是装饰

时间估算现在至少承担三件事：

- 给 `fixed_total_points` 估整次 run 时长
- 给 `*OPC?` 过早返回做 fallback
- 在长跑时判断 point 边界是否明显跑偏

所以它必须保留在 runtime 里，不能删。

## 3. raw truth 不是 point_fields

当前最终事实永远是：

- `raw/oe1022d.rall`
- `raw/oe1022d.frames.idx.jsonl`
- `segments.jsonl`
- `raw/oe1022d.frames.parsed.jsonl` 只是 companion truth，不替代前 3 项

`point_fields.jsonl` 只是当前最小字段级原标签，不是最终真相。

如果以后想单独切 point 的 raw，应该离线按：

- `segments.jsonl`
- `frame_seq_start/end`
- `raw_offset_start/end`
- `raw/oe1022d.frames.idx.jsonl`

去裁。

不要在采集线程里多写一套 per-point raw 文件。

## 4. 当前“锁零”不是物理零磁场

当前实现的是旧系统兼容的零偏电流锁定：

- 设定 `zero_offset_setpoint_a`
- 多次回读 `MEAS:CURR?`
- 得到 `locked_zero_offset_current_a`
- point 电流按：
- `locked_zero_offset_current_a + delta_current_a`

这不等于已经证明物理磁场为 0。

如果后续 agent 需要回答“原厂锁零到底做了什么、没做什么”，先读：

- [12_原厂锁零事实与边界.md](./12_原厂锁零事实与边界.md)

## 5. Laser 现在是 run 固定背景，不是 point 变量

当前 Laser 只支持：

- `on_background`
- `off_background`

它在 run 打开阶段配置好，point 期间不变，cleanup 统一关闭。

不要把 point 级 Laser on/off/power 控制偷偷塞回当前 runtime。

## 6. 当前磁场电源第一版不支持负输出

真机已经暴露出一个关键事实：

- 现有 M8812 供电链在当前接线/使用方式下不能把目标电流打到负值
- 之前带负号的磁场点，不应再被当成有效磁场验证

当前处理方式已经改成：

- `zero_offset_setpoint_a` 不允许为负
- 只要某个 point 算出负的 `target_current_a`
- `run execute` 和 `hardware verify-mag-lock` 都立即报错
- 默认 1D/2D/3D 示例网格全部改成非负值

## 7. point 边界已经不是固定 sleep

当前 `1 point = 1 SMB frequency sweep`。

point 有效窗口由：

- `sweep_started`
- `sweep_completed`

这对事件来定义，不再由 `acquisition_window_ms` 假装定义。

`acquisition_window_ms` 现在只是旧样例兼容字段。

## 8. 固定 out-dir 复跑不能再 append 旧 artifact

已经踩到过一次真实坑：

- 同一个 `--out-dir` 复跑时
- `points.jsonl / segments.jsonl / quality.jsonl / point_fields.jsonl / raw/oe1022d.rall`
- 曾经会静默 append 旧内容

这会直接污染 artifact 真值。

当前已经修正为：

- run 打开时重建本次要写的 `jsonl/raw/index` 文件
- 同一路径复跑会覆盖本次 runtime 生成的事实文件

因此：

- `runs/x_axis_1d_bounce_15min_live` 这次长跑的 `summary` 仍然有效
- 但它在修复前的 JSONL 行数被旧内容污染过，不能再拿来做“文件条数真值”

## 9. 当前 point 标签是什么

当前 point 级原标签已经包括：

- `target_b_nt`
- `baseline_current_a`
- `calibrated_delta_current_a`
- `target_current_a`
- RF sweep 参数
- 最小 `RALL` 字段解析
- PLL/overload 摘要
- quality 结果

但当前明确没有：

- 峰位
- 峰宽
- 对比度
- SNR
- 机器学习训练标签

这些全部后置到离线流程。

## 10. RALL 卡住的主因不是设备复杂，而是 runtime 热路径被我们复杂化

复盘 `RALL?` 问题时，最关键的教训是：原厂 LabVIEW 路径本来很简单。

LabVIEW / PyVISA probe 已验证的路径：

- 写 `RALL?`
- 等 `30ms`
- blocking exact read `12288B`
- 读完立即下一轮

我们曾经额外加入的内容反而制造了问题：

- `poll_interval_ms=48` 的读后等待
- first-byte deadline
- frame deadline
- zero-byte retry
- timeout 后 clear/retry
- 把 duplicate ratio 当作 point 失败条件

其中最直接的坏点是：full runtime 在 exact read 后还按 `poll_interval_ms` 等待，导致无 timeout 也会出现 `device_packet_counter delta>1`，即漏 50ms 窗口。

当前结论：

- 定长 `RALL?` 热路径必须保持 LabVIEW-like simple loop
- `poll_interval_ms` 不参与定长采集热路径
- deadline/retry 旋钮不再暴露到 runtime collector profile
- `payload[12287]` 作为 `device_packet_counter` 审计窗口连续性

## 11. 当前 `quality.timeout_count` 不是 run 级 timeout 总数

这个语义仍然保留：

- `quality.timeout_count` 只统计 point 窗口内 timeout
- point 开始前或 point 结束后的 collector timeout，不会记到该 point
- run 级 collector 总 timeout 应看 `events.jsonl` / `collector_stopped`

## 12. 当前 quality 规则按 unique RALL windows 判定最小帧数

当前 `quality` 规则是：

- `frames_total == 0` -> `failed_no_frames`
- `frames_unique < min_frames` -> `failed_min_frames`
- `point 窗口内 timeout_count > max_timeout_count` -> `failed_timeout`
- `last_frame_age_ms` 超阈值 -> `failed_quality`
- 其余 -> `passed`

`duplicate_ratio` 仍然记录，但不再单独导致失败。

原因：

- tight loop 可能比设备 50ms 窗口更新更快
- `device_packet_counter delta=0` 代表重复窗口，不是丢包
- 真正要警惕的是 `delta>1`

## 13. `frame_coverage_ratio` 仍然只是诊断字段

旧误区是把 `estimated_frames_expected = sweep_duration / poll_interval_ms` 直接当作 point 正常帧数。

现在必须区分：

- host 读循环可以比 50ms 设备窗口更快，因此会产生重复窗口
- unique RALL windows 才更接近实际有效 50ms 窗口数
- `frame_coverage_ratio` 可以帮助观察运行形态，但不能单独作为丢包或失败判据

## 14. 旧 tc100ms / monitor 组合的长跑问题已被后续 Windows PyVISA + simple loop 改造取代

历史 `tc100ms / monitor` 组合确实暴露过：

- 连续 timeout streak
- collector 不再增长
- cleanup 挂住

但该阶段仍带有已被删除的 runtime 复杂度。当前 Windows full stack 的最新验证是：

- run：`runs/win_full_x_axis_15min_simplified_20260613`
- `points_passed = 21 / 21`
- `collector_timeout_total = 0`
- `device_packet_counter delta_gt1_count = 0`
- `estimated_missing_windows = 0`
- `run audit-continuity` verdict = `continuous`

因此当前表述应改为：

- “旧实现曾因复杂的 RALL timing/retry/poll 逻辑暴露长跑失败”
- “当前实现已回到 LabVIEW-like simple loop，并通过 Windows full stack 15 分钟长测”

## 15. 手册里的 padding “空字符”假设暂时不能拿来做强校验

当前 `frames.parsed` 已经开始记录 `padding_status`，但真机 run 里绝大多数帧都表现为：

- `padding_status = non_zero`

这和“最后 3072 Bytes 是空字符”的手册转写存在张力。

第一版工程结论应收敛为：

- 先把 padding 当成诊断字段记录
- 不要立即把 `padding 非全零` 判成 parse failure
- 等后续再结合 PDF 原页和真机样本继续核对

## 16. OE1022D Channel-B 不能用错误 fixed profile 覆盖官方有效状态

2026-06-15 对拍官方 OE1022D 软件 CSV 时发现一个关键问题：

- 官方 CSV 的 `B-X` 有清楚 ODMR 特征
- rebuild run 的 `B-X` 只有小幅纹波
- 两者不是画图方式或单位缩放差异
- 当时 rebuild artifact 里 `b_input_overload_ratio = 1.0`、`b_gain_overload_ratio = 1.0`
- 但 `quality_status` 仍然是 `passed`

排查顺序必须按事实收敛：

- Laser 和 SMB100A 参数经人工确认相同
- Maynuo XYZ 当轮没有打开，不是主因
- RALL collector 连续性正常：`timeout=0`、`raw_len_bad=0`、`delta_gt1=0`
- 因此优先查 OE1022D effective state 和离线后处理

官方软件保存有效谱线后，rebuild 侧只读查询 Channel-B 状态，得到：

- `ICPLD? 2 = 0`，即 AC coupling
- `OFLTD? 2 = 7`，即 30 ms time constant
- `INOVD ? 2 = 0`
- `GNOVD ? 2 = 0`
- `*PLLD? 2 = 1`
- `FREQD? 2 = 4.99999e+02`

而旧 rebuild profile 错在：

- 写入 `input_coupling = 1`，即 DC coupling
- 写入 `time_constant_index = 9`，即 300 ms time constant

这两个值会把运行状态带离官方有效状态。尤其在 `300 ms dwell` 的 RF sweep 下，`300 ms` 时间常数过长；同时 DC coupling 可能引入直流偏置并触发无效状态。

当前修正为：

- `configs/profiles/oe1022d_run_ch_b_observed.json`
  - `input_coupling = 0`
  - `time_constant_index = 7`
- 配置生成器默认值同步为同一组
- generated OE profile 也同步为同一组
- 只读状态快照记录在：
  - `configs/observed/oe1022d_ch_b_official_software_state_2026-06-15.json`

修正后用同一组实验参数复跑：

- run：`runs/oe_official_state_probe_20260615_0125_run`
- `artifact-check = passed`
- `audit-continuity = continuous`
- `timeout=0`
- `raw_len_bad=0`
- `delta_gt1=0`
- `b_input_overload_ratio = 0.0`
- `b_gain_overload_ratio = 0.0`
- `b_pll_locked_ratio = 1.0`

信号量级恢复到官方 CSV 同量级：

- 旧过载 run：`B-X std = 1.80e-7`，`B-X span = 7.50e-7`
- 修正后 run：`B-X std = 1.88e-4`，`B-X span = 1.16e-3`
- 官方 CSV：`B-X std ≈ 1.91e-4`，`B-X span ≈ 1.18e-3`

后续规则：

- OE fixed profile 必须优先对齐“官方软件有效状态 + 真机只读读回”，不能只按手册默认值猜
- `quality_status=passed` 只证明采集链路连续，不证明信号有效
- `b_input_overload_ratio` 或 `b_gain_overload_ratio` 非 0 时，离线训练/分析必须至少标为 warn，不能当作正常谱线
- 不要为解决信号问题改 RALL 热路径；这次事实证明 collector 正常，问题在 OE effective state
