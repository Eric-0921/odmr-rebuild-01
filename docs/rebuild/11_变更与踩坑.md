# 变更与踩坑

本文集中记录当前 rebuild 已经踩实的技术事实，避免后续再掉回旧坑里。

## 1. `*OPC?` 不能单独当作 sweep 结束信号

真机已经证明：

- 当前实验室 SMB100A 上，`SWE:FREQ:EXEC` 后的 `*OPC?` 可能在几毫秒内返回
- 它会明显早于真实 sweep 结束

当前 runtime 方案：

- 先发 `*OPC?`
- 记录 `opc_wait_ms`
- 如果等待时间明显短于 `start/stop/step/dwell` 推出来的 sweep 时长
- 则退回到：
  - `estimated sweep duration + guard`

这不是临时 workaround，而是当前主运行链的一部分。

## 2. sweep 时间估算不是装饰

时间估算现在至少承担三件事：

- 给 `fixed_total_points` 估整次 run 时长
- 给 `*OPC?` 过早返回做 fallback
- 在长跑时判断 point 边界是否明显跑偏

所以它必须保留在 runtime 里，不能删。

## 3. raw truth 不是 point_fields

当前最终事实永远是：

- `raw/oe1022d.rall`
- `raw/oe1022d.frames.idx.jsonl`
- `segments.jsonl`
- `raw/oe1022d.frames.parsed.jsonl` 只是 companion truth，不替代前 3 项

`point_fields.jsonl` 只是当前最小字段级原标签，不是最终真相。

如果以后想单独切 point 的 raw，应该离线按：

- `segments.jsonl`
- `frame_seq_start/end`
- `raw_offset_start/end`
- `raw/oe1022d.frames.idx.jsonl`

去裁。

不要在采集线程里多写一套 per-point raw 文件。

## 4. 当前“锁零”不是物理零磁场

当前实现的是旧系统兼容的零偏电流锁定：

- 设定 `zero_offset_setpoint_a`
- 多次回读 `MEAS:CURR?`
- 得到 `locked_zero_offset_current_a`
- point 电流按：
  - `locked_zero_offset_current_a + delta_current_a`

这不等于已经证明物理磁场为 0。

## 5. Laser 现在是 run 固定背景，不是 point 变量

当前 Laser 只支持：

- `on_background`
- `off_background`

它在 run 打开阶段配置好，point 期间不变，cleanup 统一关闭。

不要把 point 级 Laser on/off/power 控制偷偷塞回当前 runtime。

## 6. 当前磁场电源第一版不支持负输出

真机已经暴露出一个关键事实：

- 现有 M8812 供电链在当前接线/使用方式下不能把目标电流打到负值
- 之前带负号的磁场点，不应再被当成有效磁场验证

当前处理方式已经改成：

- `zero_offset_setpoint_a` 不允许为负
- 只要某个 point 算出负的 `target_current_a`
- `run execute` 和 `hardware verify-mag-lock` 都立即报错
- 默认 1D/2D/3D 示例网格全部改成非负值

## 7. point 边界已经不是固定 sleep

当前 `1 point = 1 SMB frequency sweep`。

point 有效窗口由：

- `sweep_started`
- `sweep_completed`

这对事件来定义，不再由 `acquisition_window_ms` 假装定义。

`acquisition_window_ms` 现在只是旧样例兼容字段。

## 8. 固定 out-dir 复跑不能再 append 旧 artifact

已经踩到过一次真实坑：

- 同一个 `--out-dir` 复跑时
- `points.jsonl / segments.jsonl / quality.jsonl / point_fields.jsonl / raw/oe1022d.rall`
- 曾经会静默 append 旧内容

这会直接污染 artifact 真值。

当前已经修正为：

- run 打开时重建本次要写的 `jsonl/raw/index` 文件
- 同一路径复跑会覆盖本次 runtime 生成的事实文件

因此：

- `runs/x_axis_1d_bounce_15min_live` 这次长跑的 `summary` 仍然有效
- 但它在修复前的 JSONL 行数被旧内容污染过，不能再拿来做“文件条数真值”

## 9. 当前 point 标签是什么

当前 point 级原标签已经包括：

- `target_b_nt`
- `baseline_current_a`
- `calibrated_delta_current_a`
- `target_current_a`
- RF sweep 参数
- 最小 `RALL` 字段解析
- PLL/overload 摘要
- quality 结果

但当前明确没有：

- 峰位
- 峰宽
- 对比度
- SNR
- 机器学习训练标签

这些全部后置到离线流程。

## 10. `0-byte timeout` 还没消失，只是从“直接打崩”变成了“可恢复吞帧”

`2026-06-11` 的短 run 和 15 分钟长 run 已经说明：

- `读取 OE1022D RALL 定长帧 超时，已读取 0 字节` 仍会低频出现
- 新 collector 的改进点在于：
  - deadline 内 fast-poll 读取 `12288B`
  - timeout 后可恢复
  - `raw / frames.idx / frames.parsed` 仍保持条数对齐
- 但 timeout 仍会真实吞掉 point 帧数

长跑里的典型形态：

- 正常 point：`758 / 759` 帧
- 吃到一次 timeout 的 point：常见掉到 `737 / 738`
- 吃到多次 timeout 的 point：最低到 `717`

所以当前不能说“读取链已经完全稳定”，只能说：

- `ring buffer 截断` 这个旧根因已经基本排除
- 当前坏点主因已收敛到真实 collector / transport 抖动

## 11. 当前 `quality.timeout_count` 不是 run 级 timeout 总数

短 run 已经暴露出一个容易误读的点：

- collector 总 `timeout_count = 1`
- 但 3 个 point 的 `quality.timeout_count` 都是 `0`

原因不是统计坏了，而是当前实现语义就是：

- `quality.timeout_count` 只统计 point 窗口内 timeout
- point 开始前或 point 结束后的 collector timeout，不会记到该 point

## 12. `failed_timeout` 现在按 `max_timeout_count` 阈值工作，不再是 `timeout>0`

当前 `quality` 规则是：

- `frames_total == 0` -> `failed_no_frames`
- `frames_total < min_frames` -> `failed_min_frames`
- `point 窗口内 timeout_count > max_timeout_count` -> `failed_timeout`
- 其余再结合：
  - `duplicate_ratio`
  - `last_frame_age_ms`
  判成 `failed_quality` 或 `passed`

同时 `quality.jsonl` 还会额外给出：

- `collector_health = clean | recovered_timeout | degraded_timeout`
- `timeout_budget_remaining`

所以现在不能再把“collector 发生过恢复性 timeout”和“point 已经不可用”混成一件事。

## 13. 之前把 `757` 当作 point 正常帧数，是我们自己的估算偏差

这次 collector split 复验把一个旧误区也暴露出来了：

- 我们此前经常把 `estimated_frames_expected = sweep_duration / 48ms`
- 直接当作 point 正常帧数

但设备手册事实是：

- `RALL?` 数据每 `50ms` 更新一次

所以当前在真机上更稳定的“无 timeout point”更接近：

- `725 / 726` 帧

而不是我们自己按 `48ms` 算出来的 `757`。

这意味着：

- `frame_coverage_ratio` 目前仍然只能当诊断字段
- 在重新校正 expected frame 模型前，不能直接拿它做硬 fail

## 14. 旧 tc100ms / monitor 组合的长跑问题，不能再直接当作当前默认 observed profile 现状

需要明确区分两层事实：

- 历史 `tc100ms / monitor` 组合确实暴露过：
  - 连续 timeout streak
  - collector 不再增长
  - cleanup 挂住
- 但当前默认 `oe1022d_run_ch_b_observed` 已经在 `minimal_3point_runtime` 上完成：
  - 连续 3 次真机运行
  - `collector_timeout_total = 0`
  - 全 point `quality_status = passed`
  - `max(frame_gap_ms) ≈ 54.3 ~ 54.6ms`

因此当前更准确的表述应该是：

- “旧 profile 组合曾暴露过长跑挂死问题”
- “当前默认 observed profile 已经通过最小主线短跑验收”
- “更长时间的 observed-profile 长跑稳定性仍需要继续专项验证”

`summary` / `collector_stopped` 的 timeout，和 `quality.jsonl` 里的 point timeout，仍然是两个不同层级的事实，不能直接混为一谈。

## 12. 手册里的 padding “空字符”假设暂时不能拿来做强校验

当前 `frames.parsed` 已经开始记录 `padding_status`，但真机 run 里绝大多数帧都表现为：

- `padding_status = non_zero`

这和“最后 3072 Bytes 是空字符”的手册转写存在张力。

第一版工程结论应收敛为：

- 先把 padding 当成诊断字段记录
- 不要立即把 `padding 非全零` 判成 parse failure
- 等后续再结合 PDF 原页和真机样本继续核对
